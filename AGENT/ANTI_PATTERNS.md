# Architectural Anti-Patterns

This document lists architectural mistakes that the AI coding agent must avoid.

These rules exist to preserve the system architecture and prevent design drift during autonomous development.

The agent must read this file before implementing any code.

---

# 1. Do Not Merge Services

The architecture defines separate logical services:

- crawler
- extractor
- geocoder
- pipeline
- analytics exporter

These services must remain logically separate.

Forbidden behavior:

- placing crawler and extractor logic in the same module
- geocoding inside extractor
- analytics generation inside crawler

Even if the MVP runs inside a single container, **code structure must preserve service boundaries**.

---

# 2. Do Not Bypass the Snapshot Layer

Extraction must always use:


document_snapshots.clean_text


It must never operate directly on downloaded HTML.

Forbidden behavior:

- calling the extractor directly from crawler
- sending raw HTML to the LLM
- skipping snapshot persistence

Snapshots are a **core reproducibility mechanism**.

---

# 3. Do Not Mix Extraction and Geocoding

Extraction and geocoding are separate stages.

Extraction produces:


location_mentions


Geocoding produces:


geo_locations
document_locations


Forbidden behavior:

- calling Nominatim from the extractor
- geocoding inside LLM responses
- storing coordinates in `location_mentions`

---

# 4. Do Not Write to Foreign Tables

Each service owns specific database tables.

Crawler writes:


scp_objects
documents
document_snapshots


Extractor writes:


extraction_runs
location_mentions


Geocoder writes:


geo_locations
document_locations


Analytics writes:


bi_* tables


Services must **never write outside their ownership domain**.

---

# 5. Do Not Use Operational Tables for BI

Looker and analytics tools must **not query operational tables directly**.

Forbidden:


documents
location_mentions
geo_locations


Instead, analytics must use:


bi_documents
bi_locations
bi_document_locations


These tables are generated by the analytics exporter.

---

# 6. Do Not Store Derived Data in Source Tables

Operational tables must contain only primary facts.

Forbidden behavior:

- storing geocoded coordinates in `location_mentions`
- storing analytics fields in operational tables
- modifying snapshot text after creation

Derived data must live in derived tables.

---

# 7. Do Not Modify Historical Snapshots

Document snapshots are immutable.

Forbidden behavior:

- updating `raw_html`
- updating `clean_text`
- rewriting PDF files

If content changes, create a **new snapshot**.

---

# 8. Do Not Re-Crawl When Not Necessary

The system must support incremental processing.

Forbidden behavior:

- re-downloading all SCP pages during every pipeline run
- discarding existing snapshots unnecessarily

The crawler should fetch only:

- new documents
- changed documents
- explicitly requested documents

---

# 9. Do Not Change LLM JSON Format

The extraction prompt defines a strict JSON structure.

Forbidden behavior:

- adding new fields
- renaming fields
- changing nesting

Expected format:


{
"locations": [
{
"mention_text": "...",
"normalized_location": "...",
"precision": "...",
"relation_type": "...",
"confidence": 0.0,
"evidence_quote": "..."
}
]
}


Any changes must be explicitly approved.

---

# 10. Do Not Hardcode Infrastructure Values

Configuration must come from environment variables.

Forbidden behavior:

- hardcoding database URLs
- hardcoding API endpoints
- hardcoding file paths

Examples of required environment variables:


DATABASE_URL
OLLAMA_HOST
GEOCODER_URL


---

# 11. Do Not Implement Monolithic Files

Files should remain small and modular.

Forbidden behavior:

- 2000+ line service files
- placing multiple services in one module
- implementing pipeline orchestration inside random service modules

Preferred layout:


services/
crawler/
extractor/
geocoder/
pipeline/
analytics/


---

# 12. Do Not Stop the Pipeline on Single Errors

The system must tolerate partial failures.

Forbidden behavior:

- aborting the entire run because one document fails
- stopping extraction because one JSON response is invalid
- stopping geocoding because one location cannot be resolved

Correct behavior:

- log error
- skip item
- continue pipeline

---

# 13. Do Not Break Idempotency

Pipeline stages must be safe to rerun.

Forbidden behavior:

- inserting duplicate records without constraints
- reprocessing documents without checks
- duplicating geocoded locations

The system must support:

- rerunning extraction
- rerunning geocoding
- rebuilding analytics

without corrupting data.

---

# 14. Do Not Redesign the Architecture

The architecture is already defined in:


ARCHITECTURE.md
PIPELINE.md
SERVICES.md


The agent must **implement**, not redesign.

If architectural changes appear necessary, they must be proposed explicitly rather than implemented silently.

---
